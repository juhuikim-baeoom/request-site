# 0001. 웹 프로젝트 표준 채택 (DB·데이터·문서 관리)

## 상태

승인됨 (2026-07-11)

## 맥락

프로젝트가 접수 폼·내 요청·관리 보드·대시보드·계정 관리까지 커지면서, DB 네이밍·문서 관리 기준이 없어 다음 위험이 있었다.

- 스키마·문서·코드가 각자 진화하며 드리프트(불일치) 발생.
- 신규 컬럼/문서가 일관 기준 없이 추가되어 유지비 증가.
- 코드 변경 시 어떤 문서를 함께 고쳐야 하는지 사람이 매번 기억해야 함.

`web-project-standards`(SQL 스타일 가이드·진화적 DB 설계·Diátaxis·docs-as-code 기반 범용 표준)를 검토해 채택 후보로 삼았다.

## 결정

`web-project-standards`의 4개 표준 문서를 저장소 `docs/standards/`로 복사해 **SSOT로 채택**한다.

- 핵심 규칙 요약과 영향 매핑 표는 `CLAUDE.md`에 반영하고, 상세는 표준 문서로 링크한다.
- 문서는 Diátaxis 기반으로 재구성한다: `docs/00-overview/`(개요·SSOT 인덱스), `docs/reference/`(스키마·요구사항), `docs/adr/`, `docs/standards/`.
- 문서 동기화를 `.claude/hooks/docs-sync-reminder.js`(비차단 PostToolUse 훅)로 자동 상기시킨다.
- DB 표준은 **going-forward**로 적용한다(기존 운영 스키마 보존). 전면 표준화는 별도 로드맵으로 분리 → ADR-0002.

## 고려한 대안

- **표준 미채택(현행 유지)**: 드리프트·유지비 문제 지속. 기각.
- **DB 스키마 즉시 전면 표준화**: 한글 enum→영문, 예약어 컬럼 개명 등은 프론트엔드·RLS·트리거·뷰 전체를 동시에 깨뜨림. 운영 중 서비스에 과도한 위험. 기각(ADR-0002에서 점진 로드맵으로 대체).
- **차단형 문서 훅**: 문서 미갱신 시 커밋 차단. 강제력은 크나 작업 흐름을 막아 마찰이 큼. 비차단 리마인더로 대체.

## 결과

- 장점: 규칙의 SSOT가 `CLAUDE.md`+`docs/standards/`로 단일화. 코드 변경 시 문서 갱신이 훅으로 상기됨. 신규 DB 작업이 일관 기준을 따름.
- 단점/비용: 기존 스키마와 표준 사이의 간극이 당분간 공존(ADR-0002 로드맵으로 관리). 훅 리마인더가 소음이 될 수 있어 매핑 패턴을 프로젝트에 맞게 좁게 유지.
- 후속: 신규 소스 영역 추가 시 `CLAUDE.md` 영향 매핑 표에 행 추가.
